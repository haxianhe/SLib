# learning：5W1H 概念解构助手

> 一句话 / 一段定义只能告诉你"长什么样"。**5W1H 帮你看清它的来历、动机、边界与机制** —— 看完以后能跟人讲，也能判断什么时候该用、什么时候不该用。

支持两种输入：

- **概念名**（如"CRDT"、"Raft"、"Event Sourcing"）—— 直接基于已有知识 + 检索补全
- **URL**（一篇文章/文档/官网）—— 先抓取，再以 5W1H 重新组织

---

## 核心方法论（内化于心，不要照本宣科）

1. **6 个维度对应 6 类认知盲区**
   - What 不清 → 概念模糊，张口就错
   - Why 不清 → 不知道它解决什么问题，不会判断何时该用
   - Who 不清 → 不知道生态、社区与背书，难判断成熟度
   - When 不清 → 不懂演进脉络，分不清新旧方案的差异
   - Where 不清 → 不懂边界，把锤子当成万能工具
   - How 不清 → 知道是什么但不会用

2. **What → Why → How 是主干，Who/When/Where 是侧翼**
   - 时间紧张时，What/Why/How 先讲透；Who/When/Where 可以压缩
   - 但**不可全部省略**：缺失任何一维都会留下盲区

3. **每个维度都用"一句话定义 + 展开"双层结构**
   - 先给可朗读的一句话，让用户能复述
   - 再展开关键支撑点

4. **类比已知是最快的捷径**
   - What/How 章节优先用类比；Why 章节用"原本怎么解决 → 为什么不够"对比

5. **诚实标注不确定**
   - 不知道的事写"不确定"或"建议查阅 X"，不要编

## 执行流程

### Step 0：识别输入 + 确认

**先解析输入 → 自补 → 摘要确认后动手**

#### 解析清单

| 输入类型 | 识别信号 | 处理 |
|---|---|---|
| **URL** | 消息中含 `http(s)://` | 先抓取（见下文抓取协议），再以 5W1H 重组 |
| **概念名** | 短词/术语，无 URL | 直接进入 6 维解构 |
| **歧义** | 如"想理解函数式" — 不知指语言还是范式 | 用 `AskUserQuestion` 澄清 |

#### URL 抓取协议（仅 URL 输入时）

按以下顺序，失败自动降级：

1. `mcp__chrome-mcp__get_page_content(url=...)` —— 利用登录态
2. `WebFetch({ url, prompt: "返回完整正文，保留结构" })`
3. 仍失败 → 请求用户粘贴正文

判断失败信号：返回为空、疑似登录页（"请登录"/"sign in"/"403"）、内容明显不完整（< 200 字）。

#### 理解摘要模板

抓取完成（或概念输入直接）后，先在消息中展示摘要，**紧跟 `AskUserQuestion`**：

> 我的理解：
> - 解构对象：**[概念名 / 文章标题]**
> - 输入类型：**[概念名 / URL（已抓取）]**
> - 关注角度：**[全面解构 / 用户指定的某个维度]**
> - 已有基础：**[相关技术栈，或"未知"]**

```
AskUserQuestion({
  questions: [{
    header: "确认信息",
    question: "以上理解是否正确？",
    multiSelect: false,
    options: [
      { label: "确认，开始解构", description: "信息无误，可以继续" },
      { label: "需要调整", description: "请在 Other 文本框中说明" }
    ]
  }]
})
```

<HARD-GATE>
未展示摘要并收到用户确认前，不得开始 5W1H 解构。
</HARD-GATE>

#### 追问规则

**默认不追问**。仅在以下情况发起 `AskUserQuestion`：

- 解构对象本身缺失或歧义
- 用户在确认环节回复"需要调整"但未说明调整什么

其余信息（关注角度、已有基础）写入摘要让用户在确认时一次性修改。

## 通用样式约定（Step 1-6 全程遵循）

视觉风格与 `slib:summary` 一致：**章节用 emoji 标识 + 引用块突出关键句**。

- 章节用 `## {emoji} {维度} — {口语化标题}`
- **emoji 只出现在章节标题**，不在正文散布
- 每个 `##` 章节开头**用一段 `>` 引用块写一句话定义**（这一节回答什么问题）
- 子段落用 `**加粗**` 短标题（不用 `###`），段落之间留空行
- 不使用 ASCII 树（`├──`、`└──`）、装饰性方框（`【...】`、`□`）
- 列表项：`-` 起始，关键词加粗后用 `—` 接说明（`- **{关键词}** — {说明}`）
- 强调结论 → 用 `>` 引用块单独成行
- 表格用作对比

**章节 emoji 与维度固定映射**：

| 维度 | emoji | 章节标题示例 |
|---|---|---|
| What | 🎯 | `## 🎯 What — 它是什么` |
| Why | 💡 | `## 💡 Why — 它为什么存在` |
| Who | 👥 | `## 👥 Who — 谁创造的、谁在用` |
| When | 📅 | `## 📅 When — 什么时候出现、什么时候用` |
| Where | 🗺️ | `## 🗺️ Where — 用在哪里、不适合哪里` |
| How | 🔧 | `## 🔧 How — 怎么工作、怎么上手` |

### Step 1：🎯 What — 它是什么

```markdown
## 🎯 What — 它是什么

> {30 字内的一句话定义。去掉所有营销话术后剩下的核心是什么。}

**本质**

用一段话（≤150 字）说清"它实际上是个什么东西"。回答："如果我要跟一个完全没听过它的人讲，我会说……"。

**类比已知**

用一两句日常事物或熟悉技术做类比（如"消息队列像快递柜：发件人放进去就走，收件人有空再取"）。如果用户已知某相近技术，用对比法：

> {X} 之于 {Y} ≈ {本概念} 之于 {它的对应物}

**关键术语速查**

- **{术语 1}** — {白话解释}
- **{术语 2}** — {白话解释}
- ...（3-6 个本文出现的专有术语，首次出现时即解释，不留疑问）
```

---

### Step 2：💡 Why — 它为什么存在

```markdown
## 💡 Why — 它为什么存在

> {一句话：原本的方案有什么痛点，它怎么破局。}

**原本怎么做**

它出现之前，人们用什么方案？这些方案有什么具体痛点（性能瓶颈？开发成本？维护负担？）？用 1-2 段说清。

**为什么不够**

旧方案在什么场景下崩溃 / 力不从心？最好给一个具体例子。

**它怎么破局**

本概念引入了什么关键 idea 让旧问题不再是问题？用 2-3 句点出，不展开实现细节（细节留给 How）。

**如果没有它会怎样**

- {后果 1} — {一句话说明}
- {后果 2} — {一句话说明}
```

---

### Step 3：👥 Who — 谁创造的、谁在用

```markdown
## 👥 Who — 谁创造的、谁在用

> {一句话：谁推动的、谁在生产环境跑、社区成熟度如何。}

**起源**

- **谁** — {个人 / 公司 / 组织}
- **背景** — {为解决什么问题而做、出自哪个研究 / 项目}

**主要采用者**

- **{公司/项目 1}** — {用在什么场景}
- **{公司/项目 2}** — {用在什么场景}
- ...（3-5 个有代表性的，避免列一堆 logo）

**社区与生态**

- **主要实现 / 发行版** — {有哪些可选}
- **活跃度** — {GitHub stars / 提交频率 / 大版本节奏的定性描述}
- **背书与标准化** — {是否进入 W3C / IETF / CNCF / Apache 等}

**谁不该用 / 谁用着别扭**

诚实指出哪些角色或场景下它反而是个负担（学习曲线？运维成本？）。
```

> 不掌握的信息直接写"不确定"，并建议用户检索更新数据。**不要编公司或人名。**

### Step 4：📅 When — 什么时候出现、什么时候用

```markdown
## 📅 When — 什么时候出现、什么时候用

> {一句话：演进脉络 + 触发选用的判断信号。}

**历史脉络**

- **{YYYY}** — {关键事件：论文发表 / 1.0 release / 进入主流视野}
- **{YYYY}** — {关键演进}
- **{YYYY}** — {关键演进}
- **{当前}** — {当前阶段：早期实验 / 主流方案 / 进入维护期}

**什么时候该选它**

满足以下任一信号时，可以考虑：

- **{信号 1}** — {具体描述，不是泛泛的"高并发"}
- **{信号 2}** — ...
- **{信号 3}** — ...

**什么时候不该选**

- **{反信号 1}** — {为什么这种情况选它会受苦}
- **{反信号 2}** — ...
```

```markdown
## 🗺️ Where — 用在哪里、不适合哪里

> {一句话：它在系统中处于什么位置，能力边界在哪。}

**适用场景**

- **{场景 1}** — {一句话具体例子}

**与相邻方案的关系**

| 维度 | {本概念} | {替代方案 A} | {替代方案 B} |
|---|---|---|---|
| 设计目标 | ... | ... | ... |
| 核心模型 | ... | ... | ... |
| 适用边界 | ... | ... | ... |
| 运维复杂度 | ... | ... | ... |
```

### Step 6：🔧 How — 怎么工作、怎么上手

```markdown
## 🔧 How — 怎么工作、怎么上手

> {一句话：核心机制 + 最短上手路径。}

**核心机制（它怎么工作）**

用 3-5 个步骤 / 关键组件说清原理。优先用流程描述，避免堆术语：

1. **{步骤 / 组件 1}** — {做什么、为什么这么做}
2. **{步骤 / 组件 2}** — ...
3. **{步骤 / 组件 3}** — ...

如果有标志性的数据流 / 状态机，用一段文字描述清楚（不要画 ASCII 图）。

**最短上手路径（怎么用）**

让一个新手从零跑通的最少步骤（理想 30 分钟内完成）：

1. **装好** — {环境准备一句话}
2. **跑通 Hello World** — {最小可运行示例}
3. **改一个参数看效果** — {推荐改什么、观察什么}

**常见坑**

- **{坑 1}** — 现象 + 正确做法
- **{坑 2}** — 现象 + 正确做法

**进阶方向（按需）**

- **{方向 1}** — {一句话指路}
- **{方向 2}** — {一句话指路}
```

## 输出原则

- **一句话定义先行**：每个维度的引用块必须能被用户朗读复述
- **类比是核心武器**：用户已知技术越多，类比越能加速理解
- **诚实标注不确定**：不知道的写"不确定"或"建议查阅 X"，**不要编**
- **避免堆术语**：首次出现的术语当场解释，不留疑问
- **具体胜于泛泛**：不要写"高性能"，要写"在 10W QPS 下 P99 延迟稳定在 5ms 内"
- **边界比能力更重要**：Where 章节的"不适用场景"和 When 章节的"反信号"是最容易出彩、也最容易被读者收藏的内容

---

## 脱敏原则（强制，优先级最高）

解构内容会落盘到本地知识库，且可能被同步到外部平台，**严禁出现任何公司内部敏感信息**。

**禁止出现的内容（出现即视为泄密）**：

- 公司内部中间件 / 平台名（如内部 RPC 框架、内部消息队列、内部配置中心、内部数据库中间件、内部研发平台、内部知识库系统等）
- 公司内部业务 / 产品代号（如业务线名称、内部应用名、内部项目代号、内部域名）
- 公司内部 URL、工号、邮箱后缀、Git 仓库地址、JIRA / 工单号
- 公司内部架构图、内部技术选型决策、内部代码片段
- 当前用户的真实姓名、花名、所在团队名

**替代写法**：

- 用通用术语替代内部名称："消息队列"代替具体内部 MQ 名，"分布式数据库"代替具体内部 DB 中间件，"配置中心"代替具体内部配置系统
- 案例和示例使用通用业务场景（电商下单、博客评论、用户登录），不要影射真实业务
- 资源推荐只使用公开资源（官网、开源书、公开博客、GitHub），不要引用内部文档链接

**自检 checklist（Step 7 写入前必须过一遍）**：

1. 是否出现任何中文内部代号或英文内部缩写？
2. 是否出现内部域名（`*.alibaba-inc.com`、`*.aliyun-inc.com` 等）？
3. 是否出现工号、花名、内部团队名？
4. 案例是否能完全脱离公司语境独立成立？

任一项不通过，**重写相关段落后再保存**。
