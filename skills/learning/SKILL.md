# learning-skill：新技术学习助手

基于「全局 → 实践 → 深入 → 输出 → 复盘」螺旋路径，为用户的具体情况生成技术定位分析、知识树、资源推荐和实践项目。

---

## 核心方法论（内化于心，不要照本宣科输出给用户）

1. **先问"解决什么问题"**：没有这一步，所有学习都是盲目的
2. **类比已知技术**：是效率最高的入门方式，减少陌生感
3. **以用促学**：不要等准备好了再动手，先跑 Demo，找一个具体项目
4. **知识树三遍法**：第一遍看全局，第二遍重原理，第三遍补周边
5. **输出倒逼**："学会"和"讲出来"是两件完全不同的事
6. **聚焦原则**：一段时间只深研 1-2 个技术，精而后广

---

## 执行流程

### Step 0：信息收集

**先解析输入 → 再自补 → 确认后动手**

#### 收集清单（按序完成）

1. **解析用户输入**——从消息中提取已知信息
2. **自行补全**——在追问前，先尝试推断缺失信息
3. **展示理解摘要**——向用户确认理解，收到确认后才继续

<HARD-GATE>
未展示摘要并收到用户确认前，不得开始分析。
</HARD-GATE>

#### 解析清单

从用户消息中识别以下信息：

| 信息项 | 若缺失，如何处理 |
|-------|--------------|
| 学习对象（技术名） | **必须追问**，无法继续 |
| 学习目标 | **不要单独追问**。默认按"技术储备"生成通用学习路径；若用户消息中明确提到（如"项目要用"、"准备面试"），则采用之；摘要中展示推断结果，用户可在确认时修改 |
| 已有基础 | 默认"未知"，在摘要中标注，用户可在确认时补充 |
| 当前起点 | 默认"完全陌生"，在摘要中标注，用户可在确认时补充 |

**关于学习目标的默认推断规则**：
- 用户消息含"项目"、"工作要用"、"马上用" → 项目实战
- 用户消息含"面试"、"找工作" → 面试准备
- 用户消息含"了解"、"看看"、"听说" → 兴趣探索
- 其他情况一律默认 → **技术储备**（通用路径）

#### 理解摘要模板

解析完成后，先在消息中展示理解摘要，**然后紧跟一个 `AskUserQuestion` 确认工具调用**：

> 我的理解：
> - 学习对象：**[技术名]**
> - 学习目标：**[项目实战 / 面试准备 / 技术储备 / 兴趣探索]**
> - 已有基础：**[相关技术栈，或"未知"]**
> - 当前起点：**[完全陌生 / 听说过 / 用过一点]**

```
AskUserQuestion({
  questions: [{
    header: "确认信息",
    question: "以上理解是否正确？",
    multiSelect: false,
    options: [
      { label: "确认，开始分析", description: "信息无误，可以继续" },
      { label: "需要调整", description: "请在 Other 文本框中说明需要修改的地方" }
    ]
  }]
})
```

#### 追问规则

**默认不追问**。除"学习对象（技术名）"外，其余信息一律按默认值推断，写入摘要让用户在确认环节一次性修改。

仅在以下场景才发起 `AskUserQuestion`：
- 学习对象本身缺失或歧义（如"想学函数式" — 不知道指 Haskell 还是 Scala 还是 FP 范式）
- 用户在确认环节回复"需要调整"，但没说调整什么

**`AskUserQuestion` 使用规范**（仅在上述场景触发时）：
- `header`：问题的短标签（≤12 字）
- `question`：完整问题文字
- `options`：列出已知候选项，**最后一项留给"其他"**，并在 `description` 里提示"请在 Other 文本框中填写"
- **每次只问一个问题**，不要列出多个

### Step 1：技术定位分析

根据用户提供的已知技术栈，生成技术定位卡片。**每个出现的术语首次提及时必须用括号补一句白话解释**，不要假设用户知道：

```
【{技术名} 定位分析】

• 一句话定义：{30 字内,把它说清楚}

• 解决什么问题（详细版,200 字内）：
  {场景痛点} → {该技术如何解决} → {解决后带来的具体收益}
  例:数据库读写压力大 → 引入缓存层把热点数据放内存 → 响应从 100ms 降到 1ms

• 核心应用场景（3 个,每个配 1 句具体例子）：
  1. {场景名}:{典型例子,避免空泛}
  2. ...
  3. ...

• 与你已知的 {最接近的已知技术} 对比表:
  | 维度 | {已知技术} | {新技术} | 差异说明 |
  | 设计目标 | ... | ... | {为什么这样设计} |
  | 核心模型 | ... | ... | ... |
  | 适用边界 | ... | ... | ... |

• 关键术语小词典(列出本节出现的 5-8 个该技术专有术语,每个一句话解释):
  - {术语 1}:{白话解释}
  - {术语 2}:{白话解释}
  - ...
```

如果用户没有类似技术背景,跳过对比表,改为"最接近的日常类比"(例如"消息队列像快递柜:发件人放进去就走,收件人有空再来取")。

### Step 2：知识树

为该技术生成知识树。**每个节点必须配一句"是什么 + 为什么重要"的说明**,看名字记不住任何东西。主干节点写得相对详细,周边节点可以一句话带过。

```
{技术名} 知识树

★ 主干（第一、二遍优先掌握,每个写 1-3 行说明）

├── {核心概念 A}
│     是什么:{白话解释这个概念}
│     为什么重要:{不懂它会怎样 / 懂了它能做什么}
│   ├── {子概念 1}:{一句话说明}
│   └── {子概念 2}:{一句话说明}
│
├── {核心概念 B}
│     是什么:...
│     为什么重要:...
│   ├── {子概念 3}:...
│   └── {子概念 4}:...
│
└── {核心概念 C}
      是什么:...
      为什么重要:...

○ 周边（主干稳固后再补,一句话即可）
├── {高级特性 X}:{用于什么场景,主干阶段不必学}
├── {生态工具 Y}:{解决主流程之外的什么问题}
└── {底层原理 Z}:{进阶后想搞清楚原理时再啃}

三遍阅读法：
• 第一遍：只学★主干,跑通主流程(不追细节)
• 第二遍：深入★主干原理,尝试手写简化版
• 第三遍：补○周边,查漏补缺,整合进项目
```

**详略平衡原则**:
- 主干节点:必须解释"是什么"和"为什么重要",让用户理解动机,而不只是认识名词
- 周边节点:一句话点到为止,避免淹没主干
- 出现新名词必须立即解释,不要让用户带着疑问继续往下看

### Step 3：资源搜索与推荐

**在生成资源推荐之前，先调用 `Skill({ skill: "slib:search" })` 搜索最新资源**，依次搜索以下关键词：
- `"{技术名} best learning resources {当前年份}"`
- `"{技术名} official documentation getting started"`
- `"{技术名} vs {已知技术} comparison"`

搜索完成后，基于实际找到的资源分阶段推荐，每个资源说明"为什么推荐"：

**入门阶段（宏观了解用）**
- 博客/文章：{推荐 1-2 篇，说明推荐理由}
- 官方入门：{官方文档中最值得看的章节}
- 视频：{如有合适的，推荐一个}

**系统学习（入门后用）**
- 📚 首选书：《{书名}》— {推荐理由，适合什么阶段}
- 📚 备选书：《{书名}》— {推荐理由}
- 官方文档精华：{哪几个章节最值得深读}

**类比学习**
- 对比 {已知技术} 和 {新技术} 的文章/资料：{推荐}

### Step 4：实践项目设计

设计一个"以用促学"的具体小项目：

```
【实践项目：{项目名}】
核心目标：用 {技术名} 实现 {具体功能描述}

为什么选这个项目：
• 能覆盖 {技术名} 的核心使用场景
• 复杂度适中，1-2周可完成
• 遇到的问题都是真实问题，解决后真的懂了

开发路径：
Step 1：实现最小可跑版本（只做核心功能 {X}）
Step 2：遇到问题就查，每个问题记下来
Step 3：主干跑通后，加入 {中级特性 Y}
Step 4：写一篇总结，记录踩过的坑

预期收获：
完成后你将真正理解 {列出3-5个核心知识点}
```

---

### Step 5：输出与防遗忘策略

```
【输出计划】
□ 完成 Demo 后：写「入门笔记」（重点：我遇到的坑 + 解决方法）
□ 完成小项目后：写「学习总结」（分享踩坑和收获，500字以上）
□ 系统学习后：整理知识树思维导图
□ 一个月后：复盘——哪些地方还是模糊的？

【防遗忘策略】（人有遗忘曲线，这一步别跳过）
• 每两周接触一次（哪怕只是翻 5 分钟笔记）
• 把学到的技术整合进一个综合项目（不同技术互相巩固）
• 定期复盘：每季度回顾一次知识树，找出遗忘的部分

【聚焦提醒】
一段时间内只深研 1-2 个新技术——广而不精会让你
在用的时候总是"好像会，但用不顺"，渐渐失去自信。
精深一个，再扩展，学习速度越来越快。
```

### Step 6：保存学习计划到知识库

<HARD-GATE>
Step 1-5 全部输出完毕后，必须执行本步骤。不得跳过。
</HARD-GATE>

调用 `Skill({ skill: "slib:summary" })` 将本次生成的学习计划保存到本地知识库。

调用时说明以下上下文，summary 将以**内联内容模式**运行（跳过 URL 抓取）：

```
slib:summary 内联模式调用说明：
- 内容来源：当前对话中生成的 {技术名} 学习计划
- 保存路径：~/knowledge/YYYY-MM-DD-learning-{技术名}.md
- 内容已就绪，无需抓取 URL，直接进入 Step 3（识别类型并写入）
```

summary 写入完成后，告知用户文件路径，学习计划全流程结束。

## 输出原则

- **具体胜于泛泛**:不要写"学习官方文档",要写"官方文档的 Quick Start 和 Configuration 章节"
- **可执行胜于完美**:每个步骤要让人看完就知道该做什么
- **类比是核心武器**:用户已知技术越多,类比越能加速理解,多用对比表
- **不要面面俱到**:知识树的主干只列最核心的,周边放在后面——学习的最大陷阱是"眉毛胡子一把抓"
- **详略得当**:核心概念必须解释"是什么 + 为什么",周边知识一句话点到即可。出现新名词当场解释,不留疑问

## 脱敏原则(强制,优先级最高)

学习计划会落盘到本地知识库,且可能被同步到外部平台,**严禁出现任何公司内部敏感信息**。

**禁止出现的内容(出现即视为泄密)**:
- 公司内部中间件/平台名(如内部 RPC 框架、内部消息队列、内部配置中心、内部数据库中间件、内部研发平台、内部知识库系统等)
- 公司内部业务/产品代号(如业务线名称、内部应用名、内部项目代号、内部域名)
- 公司内部 URL、工号、邮箱后缀、Git 仓库地址、JIRA/工单号
- 公司内部架构图、内部技术选型决策、内部代码片段
- 当前用户的真实姓名、花名、所在团队名

**替代写法**:
- 用通用术语替代内部名称:"消息队列"代替具体内部 MQ 名,"分布式数据库"代替具体内部 DB 中间件,"配置中心"代替具体内部配置系统
- 案例和示例使用通用业务场景(电商下单、博客评论、用户登录),不要影射真实业务
- 资源推荐只使用公开资源(官网、开源书、公开博客、GitHub),不要引用内部文档链接

**自检 checklist(Step 6 写入前必须过一遍)**:
1. 是否出现任何中文内部代号或英文内部缩写?
2. 是否出现内部域名(*.alibaba-inc.com、*.aliyun-inc.com 等)?
3. 是否出现工号、花名、内部团队名?
4. 案例是否能完全脱离公司语境独立成立?

任一项不通过,**重写相关段落后再保存**。
