# summary：内容提炼与知识库存档

对已就绪的内容做类型识别、按类型套用模板提炼核心观点、对适合可视化的部分生成 drawio 图表，并写入本地知识库。

**本 skill 不负责内容抓取**。内容如何获取（用户粘贴、浏览器抓取、上游 skill 传入等）由调用方决定；
进入本 skill 时应已具备可直接提炼的正文。

调用方可覆盖默认信息卡字段：
- 提供字段清单（顺序、值、需省略的字段）时按调用方约定执行；未指定字段沿用默认规则
- 可指定 `类型` 为非 4 类的特殊值（如 `学习计划`、`调研报告`），此时跳过类型判断步骤

## 执行流程

### Step 0：信息收集

**先解析输入 → 再自补 → 确认后动手**

#### 解析清单

| 信息项 | 若缺失，如何处理 |
|-------|--------------|
| 正文内容 | **必须由用户/调用方提供**，无内容则无法继续 |
| 原始来源（URL / 出处） | 可空，作为信息卡 `来源` 字段；空缺时省略该字段 |

### Step 1：识别类型并提炼

#### 类型判断（按优先级匹配）

| 类型 | 判断信号 |
|------|---------|
| **技术教程** | 有步骤编号 / 代码块；标题含「如何 / 使用 / 入门 / 指南 / 教程」；以操作为主 |
| **原理架构** | 有设计讨论 / 系统对比 / 深度分析；标题含「原理 / 架构 / 设计 / 深入 / 分析」 |
| **行业资讯** | 事件驱动；含时间 / 数据 / 发布公告；主体是外部事件而非技术本身 |
| **思想观点** | 论点驱动；个人见解 / 方法论 / 反思；标题含「为什么 / 如何思考 / 我认为」 |

无法判断时默认「**原理架构**」。

#### 模板 A：技术教程

```markdown
## 🎯 功能定位

> 一句话说清这个工具/技术/框架做什么、解决什么问题。

## ⚡ 快速上手

核心操作步骤（3-5 步），关键代码片段保留：

1. **{步骤 1}** — {做什么 + 关键命令}
2. **{步骤 2}** — ...

## 🔧 核心 API / 命令

| API/命令 | 作用 | 关键参数 |
|---|---|---|
| ... | ... | ... |

## ⚠️ 常见坑

- **{坑 1}** — 现象 + 正确做法
- **{坑 2}** — ...

## 💡 适用场景

什么情况下应该用？哪些情况不适合？
```

#### 模板 B：原理架构

```markdown
## 🎯 核心设计理念

> 为什么要这样设计？解决了什么根本问题？

## 🏗️ 关键架构决策

3-5 个核心设计选择，每个写「是什么 + 为什么这样选」：

- **{决策 1}** — {内容}；选择理由：{...}
- **{决策 2}** — ...

## ⚖️ 技术权衡

**优势**

- ...

**劣势 / 局限**

- ...

## 🔄 与现有方案对比

| 维度 | 本方案 | 其他方案 |
|---|---|---|
| ... | ... | ... |

## 💡 启发与应用

这个设计理念能在自己的哪些场景中借鉴？
```

#### 模板 C：行业资讯

```markdown
## 📰 核心事件

> 2-3 句话说清楚发生了什么。

## 📊 关键数据 / 时间线

- **{YYYY-MM-DD}** — {事件节点 1}
- **{YYYY-MM-DD}** — {事件节点 2}

## 🌊 影响分析

**对行业**

- ...

**对自己 / 团队**

- ...

## 🤔 我的判断

这件事重要吗？为什么？倾向于怎么看？

## 🔮 后续值得关注

接下来应该关注什么信号或进展？
```

#### 模板 D：思想观点

```markdown
## 💭 核心论点

> 作者主要主张什么？用自己的话概括。

## 📚 支撑论据

作者用哪些论据支撑？质量如何？

- **{论据 1}** — {内容}
- **{论据 2}** — ...

## ✨ 启发与应用

读完能用在哪里？改变了什么认知？

## 🔍 反驳与局限

观点的局限性或反例？

## 💬 一句话总结

> 推荐给同事时，怎么用一句话讲清它的价值？
```

### Step 1.5：识别可视化内容并生成 drawio 图

提炼完成后扫描内容，判断是否需要图示辅助理解。**纯论点 / 单一概念 / 资讯类文章可直接跳过本步**。

#### 触发可视化的信号

| 内容特征 | 推荐图表类型 |
|---------|-------------|
| 多组件协作、模块分层、系统结构（组件 ≥ 4） | 架构图 |
| 步骤序列、决策分支、数据处理流程（步骤 ≥ 3） | 流程图 |
| 跨角色交互、API 调用次序、异步消息 | 时序图 |
| 时间线事件、阶段演进 | 时间线 |
| 概念层级、知识体系 | 思维导图 |

#### 生成与预览

1. **图必须好看,不只是"画出来"**——总结类文章最终是要被反复阅读、收藏甚至分享的,图的质感直接决定知识卡的档次。本 skill 在这步默认按「演示级 / 对外可看」标准产出,而不是草稿质量。
2. 图表样式、配色、骨架规范**强制遵循** `slib:architecture-diagram` skill 的「确认门 / 美观要求」分支:
   - **配色统一**:按角色分类,整图主色不超过 4-5 种,背景 / 节点 / 强调三层分明,禁止每节点一种色
   - **层次清晰**:同层级元素尺寸 / 字号一致,主流程在视觉中心,辅助元素弱化(灰色 / 描边)
   - **留白充足**:节点间距 ≥ 40px,避免拥挤;长标签换行,不要超出节点边界
   - **字体规范**:统一字号(标题 / 节点 / 注释三档),中文用无衬线(PingFang SC / Source Han Sans),不混用花体
   - **风格统一**:同一篇 md 内多张图共用同一套配色与圆角风格,不允许"图 1 是方角扁平、图 2 是圆角拟物"
   - 不确定时先调用 `slib:architecture-diagram` 让其产出 XML,再回到本 skill 落盘
3. **优先调 `mcp__drawio__open_drawio_xml`**(详图)或 `mcp__drawio__open_drawio_mermaid`(简图)打开预览
   - MCP 工具会优先唤起本地 drawio 桌面 app;未安装时自动降级到浏览器(diagrams.net)
   - 用户无需手动选择
4. **同时落盘**为 `.drawio` 文件,路径与 markdown 同目录:
   - 单图:`~/knowledge/YYYY-MM-DD-{title}-fig{N}-{slug}.drawio`
   - 多图按 `fig1`、`fig2`、`fig3` 顺序编号,编号顺序与正文中出现顺序一致
5. **自检一遍再交付**:打开预览后,对照下面 checklist 过一遍,任一项不达标就回到上一步重画,**不要把粗糙的图丢给用户**:
   - [ ] 一眼能看出主流程 / 主结构,不需要逐节点读字
   - [ ] 配色统一,无"彩虹节点"
   - [ ] 文字不溢出、不重叠、不被边框切掉
   - [ ] 同篇多图风格一致(配色、圆角、字号)
   - [ ] 标题、图例(如有)完整

#### 维护图表索引表（内部使用，供 Step 3 引用）

每生成一张图，记录到下表（仅在本 skill 执行期内保留，不写入 markdown）：

| 图编号 | drawio 文件路径 | 图表类型 | 一句话说明 | 对应正文章节 |
|-------|----------------|---------|-----------|------------|
| fig1 | `~/knowledge/...fig1-arch.drawio` | 架构图 | 三层系统结构 | `## 🏗️ 关键架构决策` |
| fig2 | ... | 时序图 | 调用链路 | `## 🔧 核心 API / 命令` |

### Step 2：写入本地知识库

**文件路径**：`~/knowledge/YYYY-MM-DD-{slugified-title}.md`

- 标题 slug 规则：取文章标题，空格替换为 `-`，去除特殊字符，全小写（中文标题保留原文）
- 由其他 skill 调用时建议：`~/knowledge/YYYY-MM-DD-{来源skill}-{主题}.md`（如 `learning-Rust`）

**文件格式**（**禁止使用 YAML frontmatter**，统一用信息卡）：

```markdown
# {文章标题}

> **来源** {原始链接 / 出处}  
> **日期** {YYYY-MM-DD}  
> **类型** {技术教程 / 原理架构 / 行业资讯 / 思想观点}  
> **标签** {tag1} · {tag2} · {tag3}  
> **关键概念** {concept1} · {concept2} · {concept3}

---

{提炼内容正文，在适合插图的章节段落之间留占位符}

<!-- 图 fig1：{一句话说明} | drawio: {.drawio 文件相对路径或绝对路径} -->

{后续正文}
```

**图占位符规则**：
- Step 1.5 生成的每张图都要在正文相应位置留下 HTML 注释占位符
- 占位符放在最相关章节的小标题下方、紧贴解释段落之后
- 占位符**保留 drawio 文件路径**，方便用户直接打开
- 用户后续从 drawio 编辑器导出 PNG/SVG 后，把 `<!-- ... -->` 整行替换为 `![{说明}]({图片相对路径})`

**信息卡书写规则**：
- 用单层 `>` 引用块包裹，每行末尾保留**两个空格**实现换行（Markdown 强制换行语法）
- 字段名加粗，与值之间用一个空格
- `标签`、`关键概念` 多个值之间用 ` · `（中文间隔号，前后各空格）连接，不要用方括号或逗号
- `标签` 反映文章所属领域（最多 4 个）；`关键概念` 提取文章核心术语（3-5 个，便于后续按概念检索）
- 字段顺序固定：来源 → 日期 → 类型 → 标签 → 关键概念；可省略不适用的字段，但保持此顺序
- 信息卡下方留一空行，再写 `---` 分隔线，再留空行后开始正文

写入完成后给出基础提示（详细贴图指引在 Step 3）：

> ✓ 已保存到 `{md 文件路径}`

### Step 3：输出贴图指引

**仅当 Step 1.5 生成了至少一张图时执行**。无图时跳过本步。

向用户清晰输出三件事：md 文件路径、所有 drawio 文件路径与说明、每张图对应的粘贴位置。

#### 输出模板

> ✓ **已保存总结**：`{md 文件路径}`
>
> ✓ **已生成并打开预览的图表**（drawio 编辑器已唤起）：
> - **fig1** — `{drawio 路径}`：{一句话说明}
> - **fig2** — `{drawio 路径}`：{一句话说明}
>
> 📌 **贴图指引**（请按以下顺序操作）：
>
> 1. 在已打开的 drawio 编辑器中检查 / 微调每张图
> 2. 在 drawio 中通过 `File → Export As → PNG`（或 SVG）导出图片，建议保存到 `~/knowledge/images/{md 文件名}/figN.png`
> 3. 打开 md 文件，按下表替换占位符：
>
> | 图编号 | md 中占位符位置（章节） | 替换为 |
> |-------|---------------------|--------|
> | fig1 | `## 🏗️ 关键架构决策` 章节下方 `<!-- 图 fig1: ... -->` | `![三层系统结构](./images/{md 文件名}/fig1.png)` |
> | fig2 | `## 🔧 核心 API / 命令` 章节下方 `<!-- 图 fig2: ... -->` | `![调用链路](./images/{md 文件名}/fig2.png)` |
>
> 💡 若桌面 drawio 未打开，本次预览会自动降级到浏览器 (`https://app.diagrams.net/`)，请在浏览器里完成导出。

#### 注意事项

- **不要替用户做导出/粘贴**：本 skill 只生成 drawio 源文件和占位符，导出 PNG/SVG 与替换占位符由用户在 drawio 编辑器中手动完成
- **贴图指引必须明确章节定位**：使用 md 中真实存在的小标题文本，让用户能搜索定位
- **多图须按 fig 编号列全**，不要漏图；图的顺序与正文出现顺序一致

## 质量标准

- **来源有据**：提炼内容来自给定正文，不补充原文没有的信息
- **结构精简**：每个要点一句话，不做大段复述
- **类型契合**：选定的模板章节须与文章内容匹配；若多数章节空洞，说明类型选错，重新判断
- **标签准确**：`标签` 反映领域（≤4 个），`关键概念` 反映文章独有术语（3-5 个），二者不要重复
- **emoji 仅章节标题用**：正文不要散布 emoji，保持阅读节奏
- **内容缺失透明**：若调用方/用户未提供可用正文，直接告知并请求补充，不基于猜测生成内容
- **图表必要性优先**：只为真正需要可视化的内容生图（结构 / 流程 / 时序 / 时间线 / 层级），不为单一概念或纯论点强行配图
- **图占位符必须配套贴图指引**：md 中留了 `<!-- 图 figN -->` 就必须在 Step 3 列出对应的章节定位与替换示例，二者不可缺一
- **图编号连续**：fig1 / fig2 / fig3 顺序与正文出现顺序一致，不跳号
